## Adversarial Review Protocol

After completing each PR's implementation (before tagging), you MUST spawn a
**review subagent** using the Agent tool. The reviewer is a separate agent with
fresh context — it has not seen your implementation process and has no sunk-cost
bias toward your choices.

**The reviewer's job is to find problems, not to approve.** A review that finds
nothing wrong is a failed review — it means the reviewer wasn't thorough enough.

### Review agent prompt template

Use this prompt (adjusted for PR1 vs PR2) when spawning the review agent:

```
You are a senior Go engineer reviewing a PR for correctness, completeness, and
regression risk. You are adversarial — your job is to find problems. A review
that finds zero issues is a failed review.

The PR adds a database Dialect abstraction to msgvault to support PostgreSQL
alongside SQLite. [PR1: This PR extracts existing SQLite behavior behind an
interface — zero functional change expected. / PR2: This PR adds the PostgreSQL
implementation.]

## Review checklist — you must check ALL of these:

### Completeness (grep for stragglers)
- [ ] grep -r "datetime('now')" — must find ZERO hits outside dialect_sqlite.go
- [ ] grep -r "GROUP_CONCAT" — must find ZERO hits outside dialect_sqlite.go
- [ ] grep -r "INSERT OR IGNORE" — must find ZERO hits outside dialect_sqlite.go
- [ ] grep -r "INSERT OR REPLACE" — must find ZERO hits outside dialect_sqlite.go
- [ ] grep -r "isSQLiteError" — must find ZERO hits outside dialect_sqlite.go
- [ ] grep -r "PRAGMA " — must find ZERO hits outside dialect_sqlite.go (and subset.go which is deferred)
- [ ] grep -r "messages_fts" in store/ — must only appear in dialect files and schema files
- [ ] grep -r "sqlite3\." — must find ZERO hits outside dialect_sqlite.go (and subset.go)

### Behavioral equivalence (PR1 only)
- [ ] Compare the SQL generated by SQLiteDialect methods against the original
      hardcoded SQL. Character-for-character equivalence expected.
- [ ] Verify Rebind() is still a no-op for SQLite
- [ ] Verify every isSQLiteError() call site maps to the correct dialect method
      (IsDuplicateColumnError vs IsConflictError vs IsNoSuchTableError)

### Interface design
- [ ] Can the Dialect interface be implemented by PostgreSQL without hacks?
      Look for SQLite assumptions leaking through method signatures.
- [ ] Are method signatures general enough? e.g., FTSSearchClause should not
      assume a JOIN is needed (PostgreSQL doesn't need one).
- [ ] Does the interface handle the case where FTS is unavailable? (SQLite can
      be built without FTS5; PostgreSQL always has tsvector.)

### Correctness
- [ ] Rebind() for PostgreSQL: does it handle '?' inside quoted strings correctly?
      e.g., "WHERE name = 'what?' AND id = ?" should only convert the second ?.
- [ ] FTS backfill: does the PostgreSQL batch SQL correctly join message_bodies
      and message_recipients to build the tsvector?
- [ ] GROUP_CONCAT → STRING_AGG: PostgreSQL's STRING_AGG returns NULL for empty
      sets (same as GROUP_CONCAT), but verify COALESCE wrapping is preserved.
- [ ] INSERT OR IGNORE → ON CONFLICT DO NOTHING: verify the conflict target
      columns are specified (PostgreSQL requires them).
- [ ] Error type assertions: pgx errors use pgconn.PgError with Code field,
      not string matching. Verify the dialect uses error codes, not messages.

### Edge cases
- [ ] What happens if Open() receives a postgres:// URL but pgx driver is not
      compiled in? Does it fail with a clear error?
- [ ] What happens if PostgreSQL schema init fails partway? Is it transactional?
- [ ] Are there any AUTOINCREMENT / BIGSERIAL mismatches in INSERT...RETURNING?
- [ ] Does CheckpointWAL() gracefully no-op for PostgreSQL?

### Test coverage
- [ ] Are there tests that exercise every Dialect method?
- [ ] For PR2: do tests run against both SQLite and PostgreSQL?
- [ ] Are FTS search tests included (not just FTS insert)?

### Code quality
- [ ] No dead code left behind (old functions that were replaced by dialect)
- [ ] No TODO comments that should have been resolved in this PR
- [ ] All new files have proper package documentation
- [ ] go vet and go fmt pass
```

1. Read the review findings carefully.
2. Fix ALL 🔴 BLOCKERs and all 🟡 ISSUEs.
3. Commit fixes.
4. **Spawn a second review agent** for a re-review of just the fixes.
5. Only tag when the re-review has zero BLOCKERs.
6. If the re-review finds new BLOCKERs, fix and review again. Loop until clean.

**Goal:** Move all SQLite-specific behavior behind a `Dialect` interface. Zero

### Step 2: Define the Dialect interface

Create `internal/store/dialect.go`:

```go
type Dialect interface {
    DriverName() string
    Rebind(query string) string
    Now() string
    GroupConcat(expr, sep string) string
    InsertOrIgnore(sql string) string

    // Full-text search
    FTSUpsertSQL() string
    FTSSearchClause(paramIndex int) (join, where, orderBy string)
    FTSDeleteSQL() string
    FTSBackfillBatchSQL() string
    FTSAvailable(db *sql.DB) (bool, error)
    FTSNeedsBackfill(db *sql.DB) (bool, error)
    SchemaFTS() string

    // Connection lifecycle
    InitConn(db *sql.DB) error
    SchemaFiles() []string
    CheckpointWAL(db *sql.DB) error

    // Schema migration
    SchemaStaleCheck() string
    IsDuplicateColumnError(err error) bool

    // Error handling
    IsConflictError(err error) bool
    IsNoSuchTableError(err error) bool
}
```

This is a starting point. You may need to adjust method signatures as you
discover additional call patterns while replacing call sites. That's fine —
iterate. But the interface must remain implementable by both SQLite and
PostgreSQL without hacks.

Commit: "Add Dialect interface for database backend abstraction"

### Step 3: Implement SQLiteDialect

Create `internal/store/dialect_sqlite.go`. Move `isSQLiteError()` into this
file (or make it a method). Implement every interface method to produce the
exact SQL that is currently hardcoded.

Commit: "Implement SQLiteDialect with existing SQLite behavior"

### Step 4: Wire Store to use Dialect

Add `dialect Dialect` field to `Store`. Set it in `Open()` and `OpenReadOnly()`.
Update `Rebind()`, `CheckpointWAL()`, `FTS5Available()`, `NeedsFTSBackfill()`,
`SchemaStale()`, `InitSchema()` to delegate to dialect.

Commit: "Wire Store to delegate to Dialect"

### Step 5: Replace all SQLite-isms in store package

Go through each file and replace hardcoded SQLite constructs with dialect calls:
- `datetime('now')` → `s.dialect.Now()`
- `GROUP_CONCAT(...)` → `s.dialect.GroupConcat(...)`
- `INSERT OR IGNORE` → `s.dialect.InsertOrIgnore(...)`
- `isSQLiteError(err, "duplicate column name")` → `s.dialect.IsDuplicateColumnError(err)`
- FTS5 queries → `s.dialect.FTS*()`
- etc.

**Do NOT touch `subset.go`** — it's deferred. Leave it using SQLite directly.

This is the largest step. Take it file by file:
1. `messages.go` (biggest, ~35 replacements)
2. `api.go` (~15 replacements)
3. `sources.go` (~5 replacements)
4. `sync.go` (~3 replacements)
5. `inspect.go` (if any)

Commit after each file: "Migrate {filename} to use Dialect"

### Step 6: Run tests

Run `make test`. All tests must pass. Fix any failures.

Commit fixes if needed: "Fix test failures after dialect extraction"

### Step 7: Review + QA

Spawn the review agent and QA agent as described above.
Fix any findings. Re-review until clean.

Tag: `git tag pr1-dialect-extraction`

## PR2: PostgreSQL Dialect

**Goal:** Add PostgreSQL as an opt-in backend. SQLite remains default.

### Step 1: Add pgx dependency

```bash
go get github.com/jackc/pgx/v5/stdlib
```

Commit: "Add pgx PostgreSQL driver dependency"

### Step 2: Create schema_pg.sql

Create `internal/store/schema_pg.sql`. Translate from `schema.sql`:
- `INTEGER PRIMARY KEY` → `BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY`
- `DATETIME` → `TIMESTAMPTZ`
- `BLOB` → `BYTEA`
- `BOOLEAN` → same (native in PG)
- `JSON` → `JSONB`
- Add `search_fts TSVECTOR` column on `messages`
- Add `CREATE INDEX messages_search_fts_idx ON messages USING GIN (search_fts)`
- All indexes from schema.sql, translated (partial indexes with WHERE work in PG)
- No FTS5 virtual table

Embed it: update the `//go:embed` directive in `store.go` to include
`schema_pg.sql`.

Commit: "Add PostgreSQL schema"

### Step 3: Implement PostgreSQLDialect

Create `internal/store/dialect_pg.go`:

- `DriverName()` → `"pgx"`
- `Rebind()` → convert `?` to `$1, $2, ...` (must handle quoted strings correctly)
- `Now()` → `"NOW()"`
- `GroupConcat(expr, sep)` → `fmt.Sprintf("STRING_AGG(%s, %s)", expr, sep)`
- `InsertOrIgnore(sql)` → rewrite `INSERT OR IGNORE INTO` to
  `INSERT INTO ... ON CONFLICT DO NOTHING` (must identify conflict target from
  the table's unique constraints)
- `FTSUpsertSQL()` → UPDATE messages SET search_fts = ... WHERE id = $N
- `FTSSearchClause()` → no join, `m.search_fts @@ plainto_tsquery('simple', $N)`,
  `ts_rank(m.search_fts, ...) DESC`
- `FTSBackfillBatchSQL()` → UPDATE messages m SET search_fts = ... FROM
  message_bodies mb WHERE ...
- `FTSDeleteSQL()` → `UPDATE messages SET search_fts = NULL WHERE id = $1`
  (or no-op if CASCADE handles row deletion)
- `InitConn()` → set search_path, statement_timeout, etc.
- `CheckpointWAL()` → no-op, return nil
- Error handling: use `pgconn.PgError` with SQLSTATE codes:
  - `23505` = unique_violation (IsConflictError)
  - `42701` = duplicate_column (IsDuplicateColumnError)
  - `42P01` = undefined_table (IsNoSuchTableError)

Commit: "Implement PostgreSQLDialect"

### Step 4: Update Open() for driver selection

Modify `Open()` and `OpenReadOnly()` to detect `postgres://` / `postgresql://`
URLs and use `PostgreSQLDialect` + `pgx` driver. SQLite path remains default.

Connection pool settings for PostgreSQL:
- `SetMaxOpenConns(25)` (vs SQLite's 4)
- `SetMaxIdleConns(5)`
- `SetConnMaxLifetime(5 * time.Minute)`

Commit: "Add PostgreSQL driver selection in Open()"

### Step 5: PostgreSQLEngine for query layer

Create `internal/query/postgres.go` implementing the `Engine` interface.
This is a new implementation parallel to `SQLiteEngine`, not a modification
of the existing one.

Key differences from SQLiteEngine:
- `$N` placeholders
- `STRING_AGG` instead of `GROUP_CONCAT`
- tsvector search instead of FTS5 MATCH
- `NOW()` instead of `datetime('now')`
- No sqlite_master checks

Read `internal/query/engine.go` for the interface, and `internal/query/sqlite.go`
for the reference implementation.

Commit: "Add PostgreSQLEngine implementing query.Engine"

### Step 6: Dual-backend test support

Update test helpers to support PostgreSQL:
- Check `MSGVAULT_TEST_DB` env var
- If set to a `postgres://` URL, run tests against PostgreSQL
- Default (unset) runs SQLite as before
- PostgreSQL test setup: CREATE SCHEMA with random name, run schema init,
  DROP SCHEMA CASCADE on cleanup

Commit: "Add dual-backend test support via MSGVAULT_TEST_DB"

### Step 7: Run tests (both backends if PG available)

Run `make test` (SQLite).
If a PostgreSQL instance is available in the sandbox, also run with
`MSGVAULT_TEST_DB=postgres://...`.

Commit fixes if needed.

### Step 8: Review + QA

Same adversarial review protocol as PR1, but with PR2-specific checks:
- Rebind correctness (quoted strings)
- FTS tsvector correctness
- Error code handling
- Connection pool settings
- Schema correctness (types, indexes, constraints)

Tag: `git tag pr2-postgresql-dialect`
